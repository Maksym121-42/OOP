# Завдання 1

Було використано структурний патерн Adapter для інтеграції різних платіжних провайдерів у фінансову платформу без змін існуючого коду ядра.
Ядро системи працює через уніфікований контракт IPaymentInfo з методами GetAmount(), GetCurrency() і GetStatus().

У реалізації створено окремі адаптери для кожного типу провайдера: JsonPaymentAdapter, XmlPaymentAdapter, StrangePaymentAdapter.
JsonPaymentAdapter читає поля формату amount_value, currency_code, transaction_state; XmlPaymentAdapter парсить XML (amount, currency, status) і перетворює значення у потрібні типи; StrangePaymentAdapter підлаштовує провайдера з нестандартними назвами методів та асинхронним отриманням стану.

Було змінено підхід до інтеграції: замість прямої роботи ядра з API провайдерів застосовано шар адаптерів, який приводить усі джерела даних до єдиного формату.
У консольній демонстрації всі платежі обробляються однаково через IPaymentInfo, незалежно від початкового формату провайдера.

Результат: система стала розширюваною — для підключення нового провайдера достатньо додати новий адаптер, не змінюючи код ядра.

# Завдання 2 

Було використано структурний патерн Facade для спрощення процесу оформлення замовлення в системі інтернет-магазину електроніки.
Фасад реалізовано у класі OrderFacade, який надає клієнтському коду один метод PlaceOrder() замість прямої взаємодії з усіма підсистемами.

У початковій постановці задачі клієнт повинен був окремо викликати логіку підрахунку вартості, застосування знижки, перевірки складу, оплати, бронювання та доставки, що робило код заплутаним і крихким.
У фінальній версії ця складність прихована: клієнт виконує лише один виклик до фасаду, а внутрішня послідовність дій керується всередині OrderFacade.

Було виділено окремі підсистеми: PricingService, DiscountService, InventoryService, PaymentGateway, ReservationService, DeliveryService.
Таким чином досягнуто зменшення зв’язності: зміни в окремих підсистемах (наприклад, заміна платіжного модуля) мінімально впливають на клієнтський код, оскільки він залежить лише від фасаду.

# Завдання 3 

Було використано структурний патерн Proxy Замісник для оптимізації роботи каталогу фільмів у стрімінговій системі. Проксі реалізовано у класі VideoProxy, який працює через той самий інтерфейс IVideo, що і реальний об’єкт RealVideo. Завдяки цьому клієнтський код не змінюється: він викликає однакові методи незалежно від того, чи це проксі, чи реальний об’єкт.

У початковій постановці задачі проблема полягала в тому, що детальна інформація про фільми (опис, актори, рейтинг, рекомендації) є важкою і не повинна завантажуватися для всіх елементів каталогу одразу. У фінальній версії завантаження таких даних відбувається ліниво: об’єкт RealVideo створюється лише при виклику ShowDetails(), тобто тільки коли користувач реально відкриває сторінку фільму.

Додатково у VideoProxy реалізовано:

контроль доступу до premium-контенту через перевірку _onlyForPremium;
кешування вже завантажених даних через поле _loadedVideo, щоб при повторному відкритті не виконувати повторне завантаження;
демонстрацію в консолі: перегляд каталогу (ShowCard()), спроба доступу звичайного користувача до premium-контенту, успішний доступ premium-користувача, повторний виклик з використанням кешу.

# Завдання 4
Було використано структурний патерн Composite (Компонувальник) для моделювання конфігурації корпоративного ПЗ з ієрархією модулів.
Патерн реалізовано через спільний інтерфейс IModule, який однаково підтримують і окремий модуль BasicModule, і група модулів ModulePack. Завдяки цьому клієнтський код не розрізняє, з чим працює: з одним модулем чи з підсистемою.

У початковій постановці задачі потрібно було забезпечити: побудову ієрархії модулів, підрахунок загальної вартості ліцензії, вивід структури та вмикання/вимикання функціональності.
У фінальній версії це реалізовано так:
GetLicensePrice() у ModulePack обчислює суму цін усіх вкладених компонентів, PrintTree() рекурсивно виводить дерево модулів, а Enable()/Disable() для групи поширюється на всі дочірні елементи.

# Завдання 5

Було використано структурний патерн Decorator для реалізації гнучкої системи додаткових можливостей документа у SaaS-платформі. Обраний патерн дозволяє динамічно додавати нову поведінку до об’єкта без зміни базового класу документа.

У реалізації створено базовий інтерфейс IDocument і базовий клас BasicDocument, який повертає початковий вміст файлу. Додатково введено абстрактний клас DocumentDecorator, що зберігає посилання на вкладений документ (InnerDocument) і передає виклик GetContent() далі по ланцюжку.

Функціональність розширено через конкретні декоратори: EncryptionDecorator, CompressionDecorator, WatermarkDecorator, AccessLogDecorator. Кожен декоратор додає власну поведінку (шифрування, стиснення, водяний знак, логування доступу) поверх уже існуючої, не змінюючи код BasicDocument.

# Завдання 6 

Було використано структурний патерн Bridge для переробки системи аналітики, у якій є два незалежні виміри: тип даних (фінансові показники, метрики користувачів, логи) і спосіб відображення (веб-дашборд, мобільний додаток, PDF-експорт).

У початковому підході кожна комбінація “тип даних + спосіб відображення” мала б окремий клас, що призводить до дублювання коду та швидкого збільшення кількості класів. У фінальній версії ці частини розділено:

абстракція даних: AnalyticsData та її реалізації FinancialData, UserMetricsData, LogsData;
реалізація відображення: IDisplayMethod та її реалізації WebDashboardDisplay, MobileAppDisplay, PdfExportDisplay.
Було змінено логіку побудови системи: замість жорстко прив’язаних комбінаційних класів реалізовано зв’язування під час створення об’єкта (наприклад, new FinancialData(new WebDashboardDisplay())). Це дозволяє незалежно додавати нові типи даних або нові канали відображення без зміни вже існуючого коду.

Для демонстрації роботи реалізовано простий консольний приклад: створено кілька комбінацій об’єктів і виведено результат через метод Show(). Повний застосунок не розроблявся, виконано лише частину, передбачену умовою завдання. Обраний патерн також забезпечує зручне розширення системи сторонніми розробниками через інтерфейс IDisplayMethod і базовий клас AnalyticsData.

# Завдання 7 

Було використано структурний патерн Легковаговик для оптимізації пам’яті у великому ігровому світі Minecraft.
Його вибір обґрунтований тим, що у світі є мільйони однотипних об’єктів (дуб, сосна, граніт), і дублювання однакових даних (модель, текстура, базові властивості) для кожного екземпляра призводить до зайвого споживання пам’яті.

У початковому підході кожен об’єкт повністю зберігав усі свої характеристики. У фінальній реалізації спільні (внутрішні) дані винесено в ObjectTypeData, а унікальні (зовнішні) параметри (x, y, isDestroyed) залишено в WorldObject.
Створення і повторне використання спільних об’єктів організовано через ObjectTypeFactory (кеш на базі словника), що дозволяє отримувати один і той самий Flyweight для однакових типів без повторного створення.

Для демонстрації в консолі створюється велика кількість об’єктів світу, після чого виводиться:

загальна кількість об’єктів;
кількість унікальних спільних типів;
різниця, яка показує обсяг повторного використання даних.
